# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Multi-platform video publishing automation — upload one video to Douyin, Bilibili, Xiaohongshu, Kuaishou, WeChat Channel (视频号), Baijiahao, TikTok, and YouTube in one operation. Uses Playwright (specifically `patchright`) to drive Chromium browsers that automate each platform's web upload flow.

Backend: **Flask** (REST API + SSE) with **Playwright/patchright** browser automation.
Frontend: **Vue 3 + Vite + Element Plus + Pinia**.
Database: **SQLite** at `db/database.db`.

## Architecture: Two Code Generations

The codebase has **two generations of code** coexisting. They target the same platforms but follow different conventions and are called from different entry points. **Always prefer the `uploader/` generation for new work.**

### New generation (`uploader/` + `sau_cli.py`) — preferred

- **Entry point:** `sau_cli.py` (CLI) imports directly from `uploader/*/main.py`
- **Conventions:** Proper `async`/`await`, `dataclass`es with `slots=True`, type annotations, structured logging via `_msg(emoji, text)`, `try/finally` for browser cleanup
- **Base class:** `uploader/base_video.py` → `BaseVideoUploader` (file validation, publish-date validation)
- **Platforms covered:** douyin (video + note/图文), kuaishou (video + note), xiaohongshu (video + note), tencent/视频号, youtube, bilibili
- Each uploader module exposes: `cookie_auth(account_file)`, `xxx_setup(account_file, ...)`, `XxxVideo`/`XxxNote` classes, and `*_PUBLISH_STRATEGY_*` constants (`"immediate"` / `"scheduled"`)

### Old generation (`myUtils/` + `sau_backend.py`) — legacy

- **Entry point:** `sau_backend.py` (Flask) calls `myUtils/postVideo.py`
- **Conventions:** Sync functions wrapping `asyncio.run()`, no dataclasses, thin wrappers that delegate to uploader modules
- **Files:** `myUtils/login.py` (4 login functions sharing `_get_account_file` + `_make_qrcode_callback` helpers, all delegate to uploader `*_cookie_gen`), `myUtils/auth.py` (1 `check_cookie` with platform→function mapper, delegates to uploader `cookie_auth`), `myUtils/postVideo.py` (4 post functions, all pass keyword arguments to uploader video classes)

## Two Parallel Call Paths

```
Web path:   sau_backend.py → myUtils/postVideo.py → uploader/*/main.py
CLI path:   sau_cli.py → uploader/*/main.py (direct)
```

The web path and CLI path **do not share dispatch logic**. Fixes to one must often be mirrored to the other. `sau_cli.py`'s `dispatch()` and `sau_backend.py`'s `/postVideo` route are independent platform→function routers.

## Platform Type Mapping

Used across backend, frontend, and database (`user_info.type` column):

| `type` | Platform |
|--------|----------|
| 1 | 小红书 (Xiaohongshu) |
| 2 | 视频号 (WeChat Channel / Tencent) |
| 3 | 抖音 (Douyin) |
| 4 | 快手 (Kuaishou) |

The frontend `platforms` array in `PublishCenter.vue` maps the same keys to Chinese labels. The SSE `/login` endpoint and `run_async_function` also use this mapping.

## SSE Login Flow

1. Frontend opens `GET /login?type=<n>&id=<account_name>` as an EventSource
2. Backend creates a `Queue`, stores it in module-level `active_queues[id]`, spawns a daemon thread
3. The thread runs `run_async_function(type, id, queue)` which creates an asyncio event loop, launches Playwright, navigates to the platform login page, extracts the QR code `src` (data URL), and pushes it to the queue
4. `sse_stream(queue)` polls the queue and yields `data: <msg>\n\n` to the client
5. The thread waits for a `framenavigated` event (URL change = login success), saves `storage_state` to `cookiesFile/{uuid}.json`, INSERTs into `user_info`, and pushes `"200"` to the queue

Key files: `myUtils/login.py` (4 login functions), `sau_backend.py:run_async_function` + `sse_stream`, `utils/login_qrcode.py` (QR decode/display helpers).

## Cookie File Management

- Stored as Playwright `storage_state` JSON files in `cookiesFile/` (configured by `BASE_DIR`)
- Filename format: `{uuid}.json` (generated by `uuid.uuid1()`)
- On publish: `browser.new_context(storage_state=account_file)` restores the logged-in session
- After successful publish: `context.storage_state(path=account_file)` updates the cookie (tokens may be refreshed)
- Cookie validity check: navigate to the platform's upload page and look for login UI elements (if present → cookie expired)
- There is one `cookie_auth` implementation in `uploader/*/main.py` (per-platform, 3-retry loop or single attempt, handles headless). `myUtils/auth.py` is a thin wrapper that dispatches to these by platform type.

## Database

Two tables created by `db/createTable.py`:

- **`user_info`**: `id INTEGER PK`, `type INTEGER` (platform), `filePath TEXT` (cookie filename), `userName TEXT`, `status INTEGER DEFAULT 0`
- **`file_records`**: `id INTEGER PK`, `filename TEXT`, `filesize REAL`, `upload_time DATETIME DEFAULT CURRENT_TIMESTAMP`, `file_path TEXT`

Notes: No foreign keys, no UNIQUE constraints, default journal mode (DELETE — can get "database is locked" under concurrent writes). All DB access is via sync `sqlite3`; even async route handlers use blocking calls.

## Configuration (`conf.py`)

Copy `conf.example.py` → `conf.py`. Keys:

| Key | Purpose |
|-----|---------|
| `BASE_DIR` | `Path(__file__).parent.resolve()` — project root |
| `LOCAL_CHROME_PATH` | Path to local Chrome executable (empty = use Playwright's bundled Chromium) |
| `LOCAL_CHROME_HEADLESS` | Default headless mode for `uploader/` classes |
| `DEBUG_MODE` | Default debug flag for uploaders (enables screenshots on failure) |
| `XHS_SERVER` | Internal XHS API server URL (xhs_uploader only) |
| `YT_PROXY` | Optional proxy for YouTube uploader |

## Playwright Conventions

- **Library:** `patchright` (a Playwright fork) — imports are `from patchright.async_api import ...`
- **Stealth:** Every browser context must call `await set_init_script(context)` from `utils/base_social_media.py` (injects `utils/stealth.min.js` to hide automation)
- **Browser launch:** Use `channel="chromium"` for the new generation; the old generation uses `channel="chrome"` or no channel
- **Browser cleanup:** Always close browser AND context in `try/finally`. The `async with async_playwright()` context manager only tears down the Playwright driver connection — it does NOT kill the Chromium child process
- **Login QR code:** Use helpers from `utils/login_qrcode.py` — `save_data_url_image()`, `decode_qrcode_from_path()`, `print_terminal_qrcode()`
- **Logging:** The new generation uses `_msg(emoji, text)` → `f"{emoji} {text}"` with per-platform loggers from `utils/log.py` (`douyin_logger`, `tencent_logger`, `kuaishou_logger`). The old generation uses `print()`.

## Key Utilities

- `utils/files_times.py`: `generate_schedule_time_next_day(total, per_day, times, timestamps, start_days)` — generates a list of future datetimes for scheduled publishing. **Note:** `daily_times` expects integer hours like `[6, 11, 14]`, NOT `"HH:MM"` strings
- `utils/constant.py`: `TencentZoneTypes` enum (视频号 categories) and `VideoZoneTypes` enum (Bilibili categories — mostly unused in production)
- `uploader/base_video.py`: `BaseVideoUploader` with `validate_video_file()`, `validate_image_file()`, `validate_publish_date()`. `MIN_SCHEDULE_LEAD_TIME = timedelta(hours=2)`

## Building and Running

### Backend

```bash
pip install -r requirements.txt
playwright install chromium
python db/createTable.py
python sau_backend.py          # → http://localhost:5409
```

### Frontend

```bash
cd sau_frontend
npm install
npm run dev                    # → http://localhost:5173
```

### CLI

```bash
sau douyin login --account <name>
sau douyin check --account <name>
sau douyin upload-video --account <name> --file <video> --title <title> [--tags tag1,tag2] [--schedule "YYYY-MM-DD HH:MM"]
sau skill install
```

CLI supports: `douyin`, `kuaishou`, `xiaohongshu`, `bilibili`, `tencent`, `youtube`. Each has `login`, `check`, `upload-video` (and `upload-note` for douyin/kuaishou/xiaohongshu).
